# Daily Task Workflow

## Purpose
This document defines the standard daily execution pattern for team members working inside a sprint.

## Daily Operating Principle
Every day should create measurable forward motion without expanding risk. The team should prefer consistent, reviewable progress over noisy activity.

## Start-Of-Day Routine
Each contributor begins the day by confirming:
- active sprint goal
- assigned tasks
- related standards
- related artifacts
- blocked or unblocked status
- shared-file risk for the day

## Daily Task Checklist
1. confirm today’s target task
2. re-check scope boundaries
3. identify required references in `docs/standards/` and `docs/artifacts/`
4. confirm ownership and reviewer expectations
5. execute only the planned unit of work
6. document blockers quickly
7. close the day with status, risks, and next step

## Task Execution Standard
Every task should move through this sequence:
- understand the task
- confirm scope
- gather context
- identify impacted files or documents
- execute the smallest correct change set
- verify locally where applicable
- prepare for review

## Daily Status Format
Each daily update should include:
- task name
- status: not started, in progress, blocked, in review, or done
- today’s outcome
- blocker if any
- next action

## Task Sizing Rule
Daily tasks should be small enough to:
- review in one sitting
- complete without hidden spillover
- avoid uncontrolled shared-file edits

If a task is too large for one day, split it before starting.

## Blocker Handling
If blocked:
- state the exact blocker
- state the impacted sprint goal
- state who must help unblock it
- avoid starting unrelated work just to stay busy

## Shared File Rule
Before touching a shared file:
- confirm it is necessary today
- confirm owner awareness
- narrow the exact section being changed

## End-Of-Day Routine
Each contributor ends the day by logging:
- what finished
- what remains
- any review requested
- any newly discovered risk
- tomorrow’s first step

## Daily Anti-Patterns
Do not:
- work on multiple unrelated tasks at once
- start the next sprint early without approval
- hide blockers until the end of the week
- use “almost done” as a substitute for a real status
